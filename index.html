🛡️
Three-Tier Data Architecture
Tier 1 (Always active): Manual entry + CSV upload. Works offline. Zero dependencies. This is JLIGHT's foundation — it never fails.
Tier 2 (When configured): HL7 message parsing and REDCap. Requires configuration and testing in your environment.
Tier 3 (Future / NHI-ready): FHIR R4 for e-Impilo, TIER.Net for national HIV/TB programme. Architecture is ready. Connections activate when these systems provide API access.
⚠️ Important: Tier 2 and Tier 3 connectors require configuration, testing, and formal data sharing agreements with the source system. They are architectural stubs — not plug-and-play connections. All JLIGHT intelligence functions without any Tier 2 or Tier 3 connection.

🔐 Security Architecture
How JLIGHT protects health data
Data at Rest
AES-256All local data encrypted with AES-256-GCM via Web Crypto API. Even if someone opens browser DevTools, they see ciphertext not patient data.
IndexedDBEncrypted local storage. Data persists through power cuts. Key stored in session memory — cleared when browser closes.
POPIA §19Data residency in South Africa. No patient identifiers required — participant IDs only. Explicit consent tracked per category.
Data in Transit
HTTPS/TLSAll communication encrypted in transit. TLS 1.2+ enforced on Netlify and Render. No HTTP fallback.
CORSBackend only accepts requests from jlight-health.co.za. Other origins are rejected at the server level.
Access Control
SHA-256 PINPIN hashed with SHA-256 + salt via Web Crypto API. Never stored in plaintext. Cannot be reverse-engineered.
RBACRole-based access control. Laboratory, Trial Monitor, Hospital Ops each see only their relevant pages and entry types.
Auto-timeoutSession expires after 30 minutes of inactivity. Warning at 25 minutes. Audit event logged on every sign-out.
5-attempt lockout5-minute lockout after 5 failed PIN attempts. Prevents brute force attacks.
Backend Protection
API KeyAll backend endpoints require X-JLIGHT-Key header. Configurable via environment variable on Render.
Rate limiting100 requests per minute per IP. Prevents DDoS and automated scanning attacks.
Supabase RLSRow Level Security on all database tables. Publishable anon key cannot access data without RLS policies.
For Enterprise / Government Deployment
Government and large institution deployment requires: (1) On-premise or SA-hosted server replacing Render, (2) Hardware Security Module for key management, (3) Active Directory / LDAP integration replacing PIN auth, (4) Penetration testing certification, (5) ISO 27001 compliance audit, (6) SITA approval for government systems. Contact JLIGHT for enterprise deployment package.

z-score Guide
|z| ≤ 2.0 — SATISFACTORY
Result within 2 SD of peer mean. Performance acceptable. Continue current method.
2.0 < |z| ≤ 3.0 — WARNING
Result approaching peer limits. Investigate calibration and reagent lot. Monitor closely.
|z| > 3.0 — UNSATISFACTORY
Result outside acceptable peer limits. Immediate investigation required. Raise CAPA. Notify QA manager.
Formula: z = (your result − peer mean) / peer SD
Schemes: SANAS EQA · RIQAS · EQAS · UKNEQAS · Bio-Rad Unity
